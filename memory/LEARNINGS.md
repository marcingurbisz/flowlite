# Learnings

Template:
- Date – [What happened/was tried] → Learning: [What we discovered]. Application: [How this changes our approach].

## Entries
- 2026-03-04 – Tried action overload as receiver function type (`ActionContext.(T) -> T?`) in DSL. → Learning: receiver function types can conflict with existing receiver-lambda DSL APIs (`stage(stage) { ... }`) and trigger overload ambiguity for `::action` references. Application: prefer explicit `(ActionContext, T) -> T?` action shape (or a wrapper type) in DSL public APIs that already use receiver lambdas.
- 2026-03-04 – Added showcase seeding to `startTestWebApplication`. → Learning: demo/background data generation should be feature-flagged (here `flowlite.showcase.enabled`) so it does not pollute non-web test contexts that share the same bean graph. Application: default new demo behaviors to disabled and enable only in explicit runtime modes.
- 2026-03-04 – Debugged cockpit static 404s in E2E. → Learning: mapping `/assets/**` to the dist root can resolve incorrectly; map asset paths to the actual `dist/assets` location (or use a unified static strategy) to avoid JS/CSS 404 and blank UI. Application: keep resource handlers explicit per subpath when serving Vite build output from Spring.
- 2026-03-04 – Added fixture-heavy BehaviorSpec tests for cockpit service. → Learning: data setup inside `when` blocks can be wiped by per-test cleanup hooks because each `then` executes as its own test case. Application: seed storage fixtures inside each `then` (or dedicated per-test setup) when assertions depend on mutable repository state.
- 2026-03-04 – Revisited cockpit instance summary query fan-out. → Learning: one DB call can still support separate “latest stage / latest status / latest error” derivations if SQL ranks latest rows per `(flow, instance, type)` rather than per instance only. Application: fetch per-type latest rows once and derive the three projections in-memory to reduce repository call fan-out while preserving semantics.
- 2026-03-04 – Refactored callable name inference in DSL. → Learning: naming inference logic for conditions/actions should share one fallback-aware utility to avoid drift in synthetic-lambda detection rules. Application: route both `inferConditionDescription` and `inferActionName` through shared helper `inferCallableName(value, fallback)`.
